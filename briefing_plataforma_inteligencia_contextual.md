# 2. Definição central do projeto

O projeto consiste em uma plataforma visual, self-hosted e modular para criar sistemas inteligentes baseados em contexto, memória, agentes, regras, fluxos e execução.

Cada canvas criado dentro da plataforma poderá representar um sistema inteligente diferente, como:

- cérebro operacional de uma empresa;
- agente comercial;
- copiloto jurídico;
- sistema de atendimento;
- criador de conteúdo;
- sistema de decisão;
- fluxo multiagente;
- base viva de conhecimento;
- automação cognitiva;
- app interno com IA.

A tese principal é:

> Uma plataforma visual e self-hosted para criar sistemas inteligentes orientados por contexto, onde cada canvas pode se tornar um agente, um fluxo, um cérebro operacional ou um app interno de IA.

# 3. Essência do produto

A essência não é apenas criar prompts, agentes ou automações.

A essência é permitir que o usuário construa sistemas contextuais executáveis.

Fórmula conceitual:

```text
Sistema Inteligente = Contexto + Memória + Regras + Agentes + Fluxo + Ferramentas + Output
```

A plataforma permite organizar esses elementos visualmente e transformá-los em execução real.

# 4. Problema que o projeto resolve

Empresas e profissionais estão usando IA de forma fragmentada:

- conversas soltas em chats;
- prompts perdidos;
- conhecimento espalhado em documentos, planilhas, e-mails, CRMs e ERPs;
- processos não documentados;
- decisões sem rastreabilidade;
- agentes genéricos sem contexto profundo;
- automações sem inteligência contextual;
- bases de conhecimento desconectadas da operação real.

O problema central:

> O conhecimento operacional existe, mas está disperso, desestruturado e pouco acionável por IA.

A plataforma resolve isso criando uma camada visual onde contexto, memória, agentes, fluxos e regras podem ser organizados, versionados, executados e reutilizados.

# 5. Visão de produto

A plataforma deve funcionar como um construtor visual de inteligência contextual.

Ela permite ao usuário:

1. criar canvases;
2. adicionar blocos de contexto;
3. conectar blocos entre si;
4. configurar agentes;
5. definir regras e restrições;
6. associar documentos e fontes de conhecimento;
7. compilar contexto;
8. executar agentes ou fluxos;
9. salvar histórico;
10. rastrear quais blocos influenciaram cada resposta;
11. versionar sistemas;
12. publicar canvases como mini-apps internos.

# 16. Coração operacional do produto

O núcleo funcional da plataforma é:

```text
Canvas → Context Compiler → Agent Execution → Traceable Output → Memory Update
```

## Canvas

Interface visual onde o usuário organiza os elementos.

## Context Compiler

Motor que transforma blocos conectados em contexto executável.

## Agent Execution

Execução de agentes ou fluxos de agentes.

## Traceable Output

Resposta com explicação de quais blocos, regras e fontes foram utilizados.

## Memory Update

Registro de aprendizados, decisões e histórico.

# 17. Context Compiler

O Context Compiler é o coração técnico e conceitual da plataforma.

## Responsabilidades

- identificar nós ativos;
- ler conexões;
- resolver dependências;
- aplicar prioridades;
- detectar conflitos;
- consultar conhecimento relevante;
- comprimir contexto;
- montar prompt final;
- executar modelo;
- registrar rastreabilidade;
- salvar logs;
- atualizar memória quando aplicável.

## Entrada

## Saída

- resposta;
- fontes usadas;
- blocos usados;
- tokens;
- custo estimado;
- logs;
- possíveis conflitos;

# 23. Funcionalidades diferenciais

## 23.1 Contexto modular

O usuário monta contexto como peças de Lego.

Exemplo:

```text
Contexto da empresa
+ Contexto do cliente
+ Contexto do produto
+ Agente comercial
+ Template de proposta
= Proposta personalizada
```

## 23.2 Versionamento

Cada alteração relevante pode gerar uma nova versão.

Exemplo:

- Canvas v1;
- Canvas v2;
- Canvas v3;
- Agente Comercial v1;
- Agente Comercial v2.

## 23.3 Comparação de versões

O usuário pode comparar resultados entre versões diferentes de contexto, prompt ou agente.

## 23.4 Teste A/B

Executar o mesmo input contra diferentes configurações.

## 23.5 Detector de conflito contextual

Exemplo:

Um bloco define tom descontraído.
Outro bloco define tom jurídico formal.

## 23.6 Medidor de qualidade do contexto

Possíveis métricas:

- clareza;
- coerência;
- completude;
- excesso de tokens;
- risco de conflito;
- risco de alucinação;
- aderência ao objetivo.

## 23.7 Simulação

## 23.8 Publicação como mini-app

# 31. Riscos do projeto

## 31.1 Complexidade excessiva cedo demais

Risco:

Tentar criar Figma, n8n, Obsidian, Langflow, Notion e Retool ao mesmo tempo.

Mitigação:

Começar com um fluxo simples e funcional:

```text
Criar canvas → adicionar contexto → conectar agente → executar → rastrear → salvar
```

## 31.2 Produto genérico demais

Risco:

A plataforma ser tão ampla que ninguém entende o valor imediato.

Mitigação:

Vender inicialmente um caso de uso concreto:

> Cérebro operacional para empresas.

## 31.3 Canvas bonito, mas sem execução real

Risco:

Virar apenas uma ferramenta visual.

Mitigação:

O botão “Executar” precisa estar no centro do produto.

## 31.4 Contexto demais

Risco:

O usuário apenas transfere a bagunça para o canvas.

Mitigação:

Criar hierarquia, escopo, prioridade, detector de conflito e compilador de contexto.

## 31.5 Falta de confiança empresarial

Risco:

Usuários não confiarem nas respostas dos agentes.

Mitigação:

Rastreabilidade desde o MVP.

# 32. Princípios de produto

1. Contexto é ativo.
2. Canvas deve gerar execução.
3. Todo sistema deve ser rastreável.
4. Memória precisa ter escopo.
5. Agentes devem ter limites claros.
6. O usuário deve conseguir reutilizar blocos.
7. O produto deve ser simples no uso e poderoso na composição.
8. A arquitetura deve ser genérica, mas o caso de uso inicial deve ser específico.
9. O sistema deve ajudar a reduzir bagunça contextual, não ampliá-la.
10. Todo canvas deve poder evoluir para um app.
